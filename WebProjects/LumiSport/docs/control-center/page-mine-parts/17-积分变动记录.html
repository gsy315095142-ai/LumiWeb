设计说明

  • 门店标识 — 顶部展示当前门店名称（如「朝阳店」），本页仅展示该门店下的积分变动
  • 玩法筛选栏 — 水平标签：「全部 / 🏒冰球 / 🥊拳王 / 🤺击剑」，在当前门店范围内筛选
  • 头行 — 左侧时间，右侧积分变化（+N 绿色 / −N 红色 / 0 橙色）
  • 原因行 — emoji玩法 + 结果 + 比分，如「🏒 疾速冰球 · 胜利（5:3）」
  • 余额行 — 灰色小字，显示变动后该门店下该玩法的积分值

积分计算规则

// 常量
SCORE_RATE = 20    // 每场变动基数
INIT_RATING = 100   // 初始积分
MIN_RATING = 1      // 积分下限

// 胜负结算
sum = redRating + blueRating
redDelta = SCORE_RATE × (blueRating / sum)
blueDelta = SCORE_RATE × (redRating / sum)

// 胜方 +delta，败方 -delta；平局双方 ±0

📈 胜方加分

delta = 20 × (对手积分 / 双方积分之和)

对手积分占比越高，加分越多 — 即"以弱胜强"获得更多积分奖励

📉 败方扣分

delta = 20 × (自己积分 / 双方积分之和)

自己积分占比越高，扣分越多 — 即"强输弱"扣除更多积分惩罚

  • 积分按门店独立 — 不同门店的积分各自独立计算，同一用户在不同门店的积分互不影响
  • 积分按玩法独立 — 冰球、拳王、击剑各自独立计算和存储（user_game_ratings 表按 store_id + gameType 区分）
  • 平局不变动 — 比分为平局时双方 ratingChange = 0
  • 积分下限为 1 — 扣分后不低于 MIN_RATING = 1，不会变成 0 或负数
  • 双方变动不对等 — 胜方加分与败方扣分的数值通常不相等（各自 delta 独立计算）
  • 赔率也基于积分 — 胜负盘赔率 = (1 / 预期胜率) × (1 - 10%手续费)，预期胜率 = 自己积分 / 双方积分之和

💡 举例说明

红方积分 100，蓝方积分 100（初始状态）：
sum = 200，redDelta = 20 × (100/200) = 10，blueDelta = 20 × (100/200) = 10
红方胜利 → 红方 +10 变成 110，蓝方 −10 变成 90

红方积分 150，蓝方积分 80（实力悬殊）：
sum = 230，redDelta = 20 × (80/230) ≈ 6.96，blueDelta = 20 × (150/230) ≈ 13.04
红方胜利 → 红方仅 +6.96（强者赢弱者加分少）
蓝方胜利 → 蓝方 +13.04（弱者赢强者加分多），红方 −6.96

💡 数据来源

后端结算时已记录 redRatingBefore → redRatingAfter，前端需新增接口 GET /api/player/rating-log?storeId={storeId} 聚合流水，支持 gameType 筛选参数。
表结构可复用 user_coin_logs 的模式，新建 user_rating_logs 表。
积分计算逻辑详见 player-rating.service.ts → applyReferenceRatingSettlement，常量定义见 player-rating.util.ts。
数据范围限定在当前门店，不会展示其他门店的积分变动。

🏪 门店隔离说明

积分按门店 + 玩法双重独立：同一用户在不同门店的同一玩法积分不同，变动记录仅展示当前门店数据。
参赛所得积分仅计入当场门店，不会跨店汇总或影响其他门店积分。
