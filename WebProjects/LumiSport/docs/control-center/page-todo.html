🎮一、Unity 对接

Unity 端的 ContestSyncManager 已实现竞猜后端的调用逻辑，但对接的是旧版 API。新版后端的接口路径、参数格式和认证方式均有变更，需要逐一适配。

1.1 API 路径与参数适配
1
Unity 侧 API 路径全部切换到新版后端接口
ContestSyncManager 当前调用的旧版路径需要替换为新版后端接口。以下是完整的路径对照：
功能 旧版 API（Unity 当前调用） 新版 API（需切换到）
配置比赛 /api/admin/configure → /api/player/admin/configure
开始报名 —（无旧版对应）新增 → /api/player/admin/open-betting
开启竞猜 —（无旧版对应）新增 → /api/player/admin/open-guessing
锁定赔率/开赛 /api/admin/lock → /api/player/admin/start-match
结算比赛 /api/admin/settle → /api/player/admin/end-match
退回玩法配置 —（无旧版对应）新增 → /api/player/admin/return-home
取消本轮 /api/admin/cancel → /api/player/admin/cancel-betting-round
下一轮 /api/admin/reset → /api/player/admin/next-round

💡 适配要点

除了路径变更外，部分接口的请求参数格式也可能有差异（字段名、数据结构等），需要逐个接口对照新版后端的 DTO 定义进行确认和调整。

1.2 认证方式对接
2
Unity 侧认证方式从明文 Token 改为 JWT Bearer Token
旧版 Unity 使用 adminToken 字段明文传递管理员凭证。新版后端使用 JWT Bearer Token + Admin Guard 进行鉴权。Unity 侧需要：
① 在请求头中添加 Authorization: Bearer <jwt_token>
② 实现 JWT Token 的获取、缓存与刷新逻辑
③ 处理 Token 过期时的自动重试
1.3 数据同步验证
3
选手名同步接口的数据结构兼容性确认
Unity 端的 ApplyContestPlayerNamesFromGameInfo 方法会从竞猜后端读取选手名并同步到 Unity 游戏内显示。该方法调用 /api/game 接口获取数据。需确认新版后端该接口的返回数据结构（选手名的字段路径、红蓝方标识方式等）与 Unity 侧的解析逻辑一致。
1.4 取消流程适配
4
Unity 取消流程适配：统一改用 return-home
Unity 的 OnCtrlCancelContest() 当前统一调用旧版 /api/admin/cancel。新版后端已将取消逻辑拆分为两个接口：

场景新版接口适用阶段
取消本局（退还竞猜币） cancel-betting-round 仅 betting 阶段
退回玩法配置页 return-home waiting / betting / started / settled 均可用

建议：Unity 统一改为调用 return-home，一步覆盖所有阶段，无需根据当前状态判断调用哪个接口。
5
Unity 新增 return-home 调用（退回玩法配置）
当前 Unity 没有调用 return-home 的逻辑。在新流程中，管理员从大屏返回玩法选择页时应调用此接口（收起当前场次、退还竞猜币）。需在 ContestSyncManager 中新增对应方法并在大屏 UI 操作中接入。
1.5 操作时序对齐
6
确认大屏端操作时序与新流程一致
新版后端对接口调用的前置状态有严格校验，需确认大屏端的操作时序已对齐：

操作前置状态要求需确认
start-match（开赛） 必须处于 betting 状态 如果小程序管理端尚未点击「开始报名」（还在 waiting），Unity 调用会返回错误
end-match（结算） 必须处于 started 状态 比赛进行中才能调用结算

注意：特别是 start-match，新版流程拆分了「开始报名 → 开启竞猜 → 开赛」三步，大屏端若直接调用开赛，需确保小程序端已完成前两步。
1.6 大屏与小程序双端同步

大屏（Unity Server）与小程序管理端可同时操作同一场次，需要一套同步机制防止并发冲突。以下功能项需 Server 端与小程序端协同开发。

7
Server 数据模型扩展：新增 configEpoch 与 displayIntentGameType
Server 端（竞猜后端）的会话数据模型需要新增两个字段：
① configEpoch — 配置版本号（整数），每次保存配置时 +1，用于并发冲突检测。小程序端保存时需携带当前版本号，Server 校验不一致则拒绝
② displayIntentGameType — 大屏当前玩法意向（字符串，如 iceHockey），大屏在玩法选择页切换时实时上报，小程序可据此提示管理员「大屏已切换到拳击，是否同步配置？」
8
Server 新增大屏玩法意向 API
新增接口供大屏在玩法选择页切换时调用，更新 displayIntentGameType 字段。该接口不触发保存配置，仅记录大屏当前浏览的玩法意向，供小程序端读取和提示。
9
保存配置接口增加并发控制（configEpoch 校验）
POST /api/player/admin/configure 请求体新增 baseConfigEpoch 字段。Server 端逻辑：
① 收到请求后，比对 baseConfigEpoch 与 Server 当前 configEpoch
② 一致 → 正常保存，configEpoch + 1
③ 不一致 → 返回 409 Conflict + Server 端最新会话数据，拒绝本次写入
10
小程序管理端 409 冲突处理
小程序端在调用保存配置接口时，需要处理 409 Conflict 响应：
① 用响应中返回的最新会话数据覆盖本地表单
② 弹出提示：「配置已被其他端更新，请确认后重试」
③ 管理员确认后可重新编辑并提交（携带新的 configEpoch）
11
GET session 接口扩展返回字段
管理端拉取会话数据时，接口需额外返回 configEpoch 和 displayIntentGameType 两个字段。小程序可据此实现：
① 展示大屏当前意向提示（如「大屏正在浏览拳击页面」）
② 判断大屏意向与已保存配置是否一致，不一致时给出同步提醒

🔧二、接口兼容层（可选方案）

如果 Unity 侧暂时无法全部切换到新版 API，可考虑在后端增加一层兼容路由，避免两边同时改动。两种方案二选一即可。

12
后端增加旧版 /api/admin/* 兼容路由（或 Unity 侧全部改用新版 API）
方案 A：后端加兼容层（改动较小）
在后端新增 /api/admin/configure、/api/admin/lock、/api/admin/settle、/api/admin/cancel、/api/admin/reset 五个路由，内部转发到新版 Service 方法。参数格式需做一层映射。

方案 B：Unity 侧全部改用新版（更干净）
即前述「1.1 API 路径适配」中列出的路径替换，一劳永逸，不需要维护旧版兼容代码。

建议：如果 Unity 侧改动成本可控，优先选方案 B，避免后端长期维护两套路由。

🖥️三、小程序后端落地

后端 API 与数据库迁移脚本均已完成开发，但尚未部署到生产环境，需要在真实服务器上运行并通过微信小程序审核。

13
生产环境部署
后端目前仅在本地开发环境运行，需要完成以下部署工作：
① 服务器选型与配置 — 选购云服务器，安装 Node.js 运行环境与数据库（PostgreSQL）
② HTTPS 配置 — 申请 SSL 证书并配置 HTTPS（微信小程序强制要求）
③ 域名绑定 — 注册域名、ICP 备案、解析到服务器
④ 数据库迁移 — 在生产环境执行数据库迁移脚本，创建表结构
⑤ 环境变量配置 — 生产环境的数据库连接串、JWT 密钥、微信 AppID/AppSecret 等
14
微信小程序审核与上线
需要完成以下微信平台相关操作：
① 合法域名配置 — 在微信小程序后台将后端 API 域名添加到「request 合法域名」
② 小程序类目选择 — 选择合适的服务类目（涉及竞猜/竞技内容，需确保类目合规）
③ 审核材料准备 — 提交小程序审核所需的功能说明、截图、隐私协议等
④ 合规评估 — 竞猜币系统需确保不触碰微信平台关于虚拟货币/赌博的红线
15
真机全链路回归测试
开发阶段主要在开发工具和模拟器中测试，上线前需要在真实微信环境中走通完整流程：
① 微信登录 → 选门店 → 进入赛事
② 管理员确认玩法 → 开始报名 → 选手报名 → 管理员审核
③ 管理员开启竞猜 → 观众签到领币 → 下注竞猜
④ 管理员开赛 → Unity 比赛对接 → Server 自动结算 → 竞猜赔付
⑤ 管理员操控（中止/退回/取消/下一轮等异常路径）
需在多款真机（iOS + Android 各至少 2 款）上验证界面适配与性能表现。
