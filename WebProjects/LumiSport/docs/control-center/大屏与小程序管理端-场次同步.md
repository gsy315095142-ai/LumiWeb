# 大屏玩法选择与小程序管理端场次配置：方案 C 同步说明（评审稿）

## 1. 文档目的与读者

| 项 | 说明 |
|----|------|
| **目的** | 约定 **Server 大屏（玩法选择页）** 与 **微信小程序管理端场次管理** 可同时操作同一门店场次时的 **数据模型、接口形态与冲突策略**，供前后端及大屏客户端对齐实现。 |
| **读者** | Server 负责人、大屏客户端负责人、小程序负责人 |
| **状态** | 评审稿 — 文中字段名、路径、错误码均可与 Server 最终设计合并或改名，以 **单一后端文档** 为准后再改小程序类型定义。 |

---

## 2. 背景与术语

### 2.1 业务事实

- **大屏「首页」** 即 **玩法选择页面**（冰球 / 拳王 / 击剑等），是门店在无后续流程或处于可配置阶段时的 **默认着陆 UI**。
- **小程序管理端** 可对同一场次执行：**保存配置**、**开始报名**、**正式开始比赛** 等，推进 `waiting → betting → started → settled`（与现有 [player-api](../API/player-api.md) 思路一致）。

### 2.2 方案 C 的含义（本稿约定）

- **大屏端** 与 **小程序管理端** 均可 **写入** 与「本场玩什么、配置意向」相关的信息。
- 双方必须依赖 **Server 上同一份持久化状态** 与 **清晰的并发规则**，避免「屏上选了 A、手机里保存了 B」长期不一致且无感知。

---

## 3. 设计原则

1. **Server 为唯一真理源**：任何一端的展示与可操作范围，以 Server 返回的会话数据为准。
2. **区分「意向」与「已确认配置」**（见下节）：大屏上的选择不一定立刻等于已落库的整场参数；小程序「保存配置」应与服务端规则一致。
3. **显式并发控制**：对可能被双端同时修改的资源，使用 **版本号 / epoch**（或等价机制），拒绝陈旧写入并返回最新数据引用。
4. **阶段闸门**：在 `betting` / `started` 等阶段，应禁止或严格限制玩法类字段的修改（与现网产品一致的可延续）。

---

## 4. 推荐数据概念（需 Server 确认字段名与落库位置）

以下为 **逻辑概念**；实现上可合并为 JSON 列、拆表或沿用 `game_sessions` 扩展字段，由 Server 评审选定。

| 概念 | 含义 | 说明 |
|------|------|------|
| **已保存场次配置** | 已通过管理流持久化、可与 `status` 一起驱动业务的配置 | 对应现有 `gameType`、`matchName`、`isMasterMode` 等；**开报名、开赛** 应依赖此套数据（或与其等价的 Server 内部状态）。 |
| **大屏玩法意向** | 大屏在玩法选择页上 **当前选中 / 高亮** 的玩法，可能尚未经小程序「保存配置」写入已保存配置 | 大屏操作主要更新此意向；小程序拉会话时应能读到，用于提示或与表单合并。 |
| **配置版本 `configEpoch`** | 单调递增整数（或 ISO 时间戳 + 校验），每次 **已保存场次配置** 或 **大屏意向** 的权威更新成功即递增 | 写入接口携带「基于哪一版修改」，用于冲突检测。 |

### 4.1 与现模型关系（建议）

- **已保存场次配置** 与当前 `PlayerSessionPayload` 中的 `gameType`、`matchName`、`isMasterMode` 对齐。
- **大屏玩法意向** 建议单独字段（例：`displayIntentGameType`），类型与 `gameType` 相同枚举：`hockey` \| `boxing` \| `fencing`。
- 是否在没有会话时也要有大屏意向：由 Server 决定（例如按 **门店 / 设备** 维度存 `lastDisplayIntent`，无 `game_sessions` 时仍返回给大屏与即将创建的首局）。

## 5. API 约定（供核对清单）

以下路径沿用现有管理端前缀习惯（如 `/api/player/admin/...`）；若 Server 统一放到其他前缀，本文仅保留 **语义清单**。

### 5.1 GET 管理端会话（或等价聚合查询）

**现状参考**：`GET /api/player/admin/session`

**建议扩展响应 `data`**（字段名为示例，可改名但需在多端统一）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `configEpoch` | number | 当前配置线版本，任意权威更新 +1 |
| `displayIntentGameType` | `GameType` | 大屏当前玩法意向 |
| 现有字段 | — | `sessionId`、`status`、`gameType`、`matchName`、`isMasterMode` 等保持不变 |

**语义建议**：

- 当 **`displayIntentGameType` 与 `gameType` 不一致** 且 `status === 'waiting'`：表示 **意向与已保存配置未对齐**，小程序可提示运营「大屏已选 X，已保存玩法为 Y」。
- Server 可选择在无正式会话时返回 `sessionId: 0` 占位（与现小程序对「暂无场次」的解析兼容），同时仍返回 `displayIntentGameType` 与 `configEpoch`。

### 5.2 POST 保存配置（小程序「保存配置」）

**现状参考**：`POST /api/player/admin/configure`

**请求体建议扩展**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `matchName` | string | 否 | 同现网 |
| `gameType` | `GameType` | 否 | 同现网 |
| `isMasterMode` | boolean | 否 | 同现网 |
| `baseConfigEpoch` | number | **建议必填** | 客户端上次 GET 得到的 `configEpoch`；与 Server 当前不一致则 **409 冲突** |

**成功响应**：返回完整会话对象（含更新后的 `gameType`、`configEpoch`）。

**业务规则（需产品 + Server 拍板）**：

- **方案 C-1（拉齐）**：保存成功时，将 **`displayIntentGameType` 更新为与本次 `gameType` 一致**。
- **方案 C-2（不自动覆盖大屏）**：保存仅更新已保存配置，大屏意向保留，由大屏自行刷新；**不一致状态** 靠 GET 与 UI 提示解决。  

（推荐 **C-1** 减少双端撕扯；若以大屏为主宰则可选 C-2。）

### 5.3 大屏更新玩法意向（Server 需提供）

建议独立接口（路径待定），语义示例：

- **方法**：`POST`
- **路径示例**：`/api/venue/display/intent` 或纳入现有大屏 WebSocket / 内部 API
- **请求体**：`{ "gameType": "hockey", "baseConfigEpoch": 12 }`（是否要求 epoch 由 Server 决定）
- **效果**：更新 `displayIntentGameType`，并 **递增 `configEpoch`**（若采用单一条版本线）

大屏仅在选择页变更玩法时调用；**不得绕过 Server** 仅改本地路由。

### 5.4 POST 开始报名（及其他阶段推进）

**现状参考**：`POST /api/player/admin/open-betting`

**建议**：

- Server 侧校验：**已保存场次配置** 合法（例如 `gameType` 已定义、门店状态允许）。
- 可选：请求体或前置条件要求 **`displayIntentGameType` 与 `gameType` 一致**，否则返回 **4xx + 明确文案**（避免未保存表单就开报名）。
- 小程序可在 **调用前** 检测本地表单与上次 GET 是否一致，按需先调 `configure`（实现细节由小程序迭代，**不以本文为准**）。

## 6. 冲突处理（HTTP 与客户端行为）

| 场景 | 建议 HTTP | 响应体建议 |
|------|-----------|------------|
| `baseConfigEpoch` 与 Server 当前不符 | `409 Conflict` | `code` + `message` + **完整最新 `data` 会话**（含 `configEpoch`） |
| 当前 `status` 不允许改玩法 | `400 Bad Request` | 明确错误码，如 `SESSION_STATUS_NOT_EDITABLE` |

小程序侧预期行为（可与 Server 同事对齐验收标准）：

- 收到 409：用响应内最新会话 **覆盖本地表单与 session**，并 Toast「配置已被其他端更新，请确认后重试」。

---

## 7. 大屏 UI 与阶段映射（产品对齐）

| Server `status`（示例） | 大屏建议行为 |
|-------------------------|--------------|
| 无场次 / `waiting`（可配置） | 可显示 **玩法选择页**；意向与已保存配置差异按 Server 数据高亮或提示 |
| `betting` | 进入 **报名中** 对应 UI，一般 **不允许** 再切换玩法（与现小程序提示一致） |
| `started` / `settled` | 进入对应赛况 / 结算 UI |

具体路由表由大屏与 Server 联合定稿；本文只约束 **状态来源唯一**。

---

## 8. 实时性（可选）

| 方式 | 说明 |
|------|------|
| 轮询 | 管理端进入页面 / 下拉刷新时 `GET .../session` |
| WebSocket / Server-Sent Events | Server 在 `configEpoch` 变更时推送，大屏与小程序订阅后刷新 |

是否必须实时：**非必须** 时，**轮询 + 409** 已可满足方案 C 一致性；实时为体验增强。

## 9. 开放问题（评审时请逐项勾选）

- [ ] `displayIntentGameType` 是否 **按门店**、**按设备**、还是 **全局单实例**？
- [ ] `configEpoch` 是一条线，还是 **意向** 与 **已保存配置** 各一条版本号？
- [ ] 无 `game_sessions` 行时，大屏意向存哪、GET 管理端 session 如何组装？
- [ ] 保存配置成功后，大屏意向是否与 `gameType` **强制对齐**（§5.2 C-1 vs C-2）？
- [ ] `open-betting` 是否强制 **意向与已保存玩法一致**？
