🔄轮询机制详解

弹幕采用 HTTP 短轮询（非 WebSocket），以 afterId 实现增量拉取：

阶段行为拉取数量
首次进入 afterId=0，拉取该场次最近 N 条（按 id DESC 取再反转为时间正序） limit=50，前端只展示最后 8 条（避免刷屏）
增量轮询 每 2.8 秒 请求一次，where id > afterId，按 id ASC limit=50，全部展示
发送成功 POST 成功后立即更新 afterId，自己的弹幕立即飘屏（不等轮询） 1 条
切换场次 重置 afterId=0，清空弹幕数组，重新首次拉取 —
离开页面 onUnmounted 停止轮询计时器，释放资源 —

💡 为什么用轮询而不是 WebSocket？

弹幕消息频率较低（非高并发直播场景），2.8 秒轮询延迟在可接受范围内。HTTP 轮询的优势：实现简单、无需维护长连接、兼容性好（小程序 WebSocket 在后台切回时可能断连需额外处理）。如果未来弹幕量增大，可平滑升级为 WebSocket 方案。
