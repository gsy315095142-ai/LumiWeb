## 1. 模块定位

弹幕互动是赛事直播场景下的**实时文字互动系统**，允许已登录用户在当前场次中以滚动飘屏的方式发送和查看弹幕消息，增强观赛氛围和社交参与感。

**核心特征**：
- 场次级隔离：每条弹幕绑定一个 `sessionId`，不同场次之间互不可见
- 轮询增量拉取：前端每 2.8 秒通过 `afterId` 增量拉取新弹幕（非 WebSocket）
- 敏感词过滤：服务端子串匹配 + 大词库（1000+ 条），命中即拒绝
- 单条内容限制：最多 80 字符

---

## 2. 角色与权限

| 角色 | 发送弹幕 | 查看弹幕 | 备注 |
|------|---------|---------|------|
| 已登录用户（user） | ✔ 需 JWT 鉴权 | ✔ 需 JWT 鉴权 | 登录后即可发送，无需其他条件 |
| 管理员（admin） | ✔ | ✔ | 同用户权限 |
| 未登录访客 | ✖ | ✖ | 接口由 `JwtAuthGuard` 拦截 |

**关键设计**：
- 弹幕功能**不限制场次状态**（waiting / betting / started / settled 均可发送）
- 弹幕功能**不限制用户角色**（选手、观众、管理员均可）
- 弹幕功能**不检查用户是否参与竞猜或报名**
- 仅依赖 JWT 认证 + 场次存在性校验

## 3. 发送弹幕

### 3.1 接口定义

```
POST /api/danmaku
Authorization: Bearer <jwt>
```

**请求体**：
| 字段 | 类型 | 必填 | 校验规则 | 说明 |
|------|------|------|---------|------|
| sessionId | number | ✔ | 整数，≥1 | 目标场次 ID |
| content | string | ✔ | 非空，≤80 字符 | 弹幕正文 |

### 3.2 服务端处理流程

```
1. 内容预处理：trim() + 合并连续空白为单个空格
2. 空内容校验 → 400 "弹幕内容不能为空"
3. 长度校验（80字符）→ 400 "弹幕内容过长（最多80字）"
4. 敏感词检测（子串匹配，不区分 ASCII 大小写）
   └─ 命中 → 400 "弹幕内容包含敏感词，请修改后重试"
5. 场次存在性校验 → 404 "场次不存在"
6. 用户存在性校验 → 404 "用户不存在"
7. 入库（session_danmaku 表）
8. 返回完整消息结构（含 id、displayName、createdAt）
```

### 3.3 内容校验规则

| 规则 | 具体实现 | 错误码 |
|------|---------|--------|
| 空内容 | `content.trim()` 后为空字符串 | 400 |
| 超长内容 | `content.length > 80`（trim + 合并空白后） | 400 |
| 敏感词 | 子串匹配，按词长降序，ASCII 不区分大小写 | 400 |
| 场次不存在 | `sessionId` 在 `game_sessions` 中无记录 | 404 |
| 用户不存在 | `userId` 在 `users` 中无记录 | 404 |

### 3.4 敏感词过滤机制

#### 词库加载

敏感词词库支持**双来源**合并（启动时加载，去重后按长度降序排列）：

| 来源 | 配置方式 | 说明 |
|------|---------|------|
| 词库文件 | `DANMAKU_SENSITIVE_WORDS_FILE` 环境变量 | 默认路径 `config/danmaku-sensitive-words.txt`，每行一词，`#` 开头为注释 |
| 环境变量 | `DANMAKU_SENSITIVE_WORDS` 环境变量 | 逗号分隔的内联词表 |

**加载优先级**：文件词表 + 环境变量词表合并去重。
**特殊配置**：`DANMAKU_SENSITIVE_WORDS_FILE=""`（显式设空）可关闭文件词库，仅用环境变量。
**空词库行为**：词库为空时不拦截任何内容（便于本地开发），生产环境应配置词库。

#### 匹配算法

```typescript
// 对每个敏感词 w（按长度降序）：
// 1. 原文子串匹配：haystack.includes(w)
// 2. 若 w 仅含 ASCII 字符，额外做不区分大小写匹配：
//    lowerAsciiFold(haystack).includes(lowerAsciiFold(w))
```

**特点**：
- 子串匹配（非全词匹配），"abc" 能匹配 "xxxabcxxx"
- 中文词不做大小写处理（中文无大小写）
- 英文词做 ASCII 大小写折叠（A-Z → a-z），绕过大小写变体
- 按词长度降序匹配，优先命中长词，减少短词误报

#### 词库来源

生产词库文件 `config/danmaku-sensitive-words.txt`，来源：
- 开源项目 Konsheng/Sensitive-lexicon（色情词库、暴恐词库、涉枪涉爆、补充词库）
- 本地历史词条
- 自动扩充时间：2026-03-30

## 4. 拉取弹幕（增量轮询）

### 4.1 接口定义

```
GET /api/danmaku?sessionId={id}&afterId={id}&limit={n}
Authorization: Bearer <jwt>
```

| 参数 | 类型 | 必填 | 默认值 | 校验规则 | 说明 |
|------|------|------|--------|---------|------|
| sessionId | number | ✔ | - | 整数，≥1 | 目标场次 ID |
| afterId | number | ✖ | 0 | 整数，≥0 | 增量游标：只返回 id > afterId 的记录 |
| limit | number | ✖ | 50 | 整数 1~100 | 单次最多拉取条数 |

### 4.2 两种拉取模式

| 模式 | afterId 值 | 行为 | 用途 |
|------|-----------|------|------|
| **首次拉取** | `0`（默认） | 返回该场次最近 N 条弹幕（按 id DESC 取 limit 条，再反转为时间正序） | 用户刚进入页面时加载历史弹幕 |
| **增量拉取** | `> 0` | 返回 id > afterId 的弹幕（按 id ASC，取 limit 条） | 定时轮询获取新弹幕 |

### 4.3 返回结构

```typescript
{
  messages: DanmakuMessagePayload[]  // 弹幕消息列表
  maxId: number                      // 当前列表中最大 id，客户端用于更新 afterId
}
```

**单条消息结构**：
| 字段 | 类型 | 说明 |
|------|------|------|
| id | number | 弹幕唯一 ID（自增主键） |
| sessionId | number | 所属场次 ID |
| userId | number | 发送者用户 ID |
| displayName | string | 发送者显示名称（通过 `playerDisplayName()` 函数获取，优先昵称，兜底邮箱前缀） |
| content | string | 弹幕正文 |
| createdAt | string | 发送时间 ISO 8601 |

### 4.4 maxId 计算逻辑

```
maxId = Math.max(afterId, ...messages.map(m => m.id))
```

客户端下次轮询时将此 `maxId` 作为新的 `afterId` 传入，实现增量拉取。

## 5. 前端组件

### 5.1 组件结构

前端弹幕功能由 `SessionDanmakuLayer.vue` 单组件实现，包含两个视觉区域：

```
┌─────────────────────────────────────┐
│      飘屏区域（dm-fly-zone）         │
│  弹幕滚动展示                       │
│  [用户A：加油！]   [用户B：厉害]     │
│        [用户C：红方必胜]             │
├─────────────────────────────────────┤
│  [发条弹幕...]         [发送]       │
│  底部输入栏（dm-input-bar）         │
└─────────────────────────────────────┘
```

### 5.2 飘屏渲染机制

| 参数 | 值 | 说明 |
|------|-----|------|
| 泳道数量 | 3 条 | 弹幕在 3 条泳道间轮转 |
| 飞行时长 | 11~16 秒（随机） | `11 + Math.random() * 5` |
| 同时在屏上限 | 16 条 | 超过后截取最后 16 条 |
| 最大宽度 | 90% 屏幕宽度 | 单条弹幕不超屏幕 90% |
| 动画方向 | 从右到左 | `translateX(750rpx) → translateX(-120%)` |
| 透明度变化 | 0.95 → 0.75 | 淡出效果 |

**泳道分配**：`lane = laneRound % 3`，循环使用 0/1/2 三条泳道。
**移除时机**：每条弹幕在 `16000ms` 后自动从列表中移除（即使动画未播完）。

### 5.3 飘屏布局模式

支持两种布局模式，通过 `flyAnchorInline` 属性切换：

| 模式 | 属性值 | 定位方式 | 适用场景 |
|------|--------|---------|---------|
| Fixed 全屏模式 | `false`（默认） | `position: fixed`，从导航栏下方开始 | 全屏弹幕覆盖 |
| Anchor 内嵌模式 | `true` | `position: absolute`，铺满父级容器 | 嵌入赛事模块，不遮挡底部 Tab |

**Fixed 模式定位计算**：
```
top = statusBarPx + navBarPx(44)
height = flyLayerHeightPx（默认 200px）
```

**输入栏**：始终 `position: fixed` 在屏幕底部，z-index 95，带安全区适配。

### 5.4 轮询策略

| 参数 | 值 | 说明 |
|------|-----|------|
| 轮询间隔 | 2800ms（2.8 秒） | `setInterval` 定时拉取 |
| 单次拉取条数 | 50 条 | `limit` 参数 |
| 首次加载截断 | 最多展示 8 条 | 首次拉取超过 8 条时只取最后 8 条 |
| 轮询失败 | 静默忽略 | 不弹 toast，不中断轮询 |

### 5.5 生命周期管理

| 事件 | 行为 |
|------|------|
| 组件挂载 | 若 `sessionId > 0`，立即首次拉取 + 开启轮询 |
| `sessionId` 变化 | 重置 `afterId=0`、清空飘屏、重新开始轮询 |
| `sessionId` 变为 0 | 停止轮询 |
| 组件卸载 | 停止轮询（`clearInterval`） |

### 5.6 发送交互

| 行为 | 说明 |
|------|------|
| 输入框 | `maxlength="80"`，`placeholder="发条弹幕"` |
| 发送按钮 | 内容为空或正在发送时 disabled（`opacity: 0.45`） |
| 键盘确认 | `confirm-type="send"`，回车触发发送 |
| 发送成功 | 清空输入框、更新 afterId、立即展示该条弹幕 |
| 发送失败 | 错误由全局 request 工具统一 toast 提示 |

### 5.7 组件 Props

| 属性 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| sessionId | number | ✔ | - | 当前场次 ID，≤0 时禁用轮询 |
| statusBarPx | number | ✔ | - | 状态栏高度(px)，用于 fixed 模式定位 |
| flyLayerHeightPx | number | ✖ | 200 | 飘屏区域高度(px)，仅 fixed 模式 |
| flyAnchorInline | boolean | ✖ | false | 是否使用 anchor 内嵌模式 |

## 6. 数据模型

### 6.1 session_danmaku 表

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | INT | PK, AUTO_INCREMENT | 弹幕唯一 ID |
| session_id | INT | NOT NULL, FK → game_sessions.id | 所属场次，`ON DELETE CASCADE` |
| user_id | INT | NOT NULL, FK → users.id | 发送者，`ON DELETE CASCADE` |
| content | VARCHAR(120) | NOT NULL | 弹幕正文（DB 列宽 120，业务限制 80 字符） |
| created_at | DATETIME(6) | AUTO, NOT NULL | 创建时间（毫秒精度） |

**外键行为**：
- 场次删除 → 该场次所有弹幕级联删除
- 用户删除 → 该用户所有弹幕级联删除

**索引建议**（基于查询模式推断）：
- `idx_session_id` on `session_id`：按场次查询的主索引
- `idx_session_id_id` on `(session_id, id)`：增量拉取的复合查询条件

### 6.2 实体关系

```
User (1) ──── (N) SessionDanmaku (N) ──── (1) GameSession
                   ↑               弹幕属于一个用户               弹幕属于一个场次
```

## 8. 错误码

| HTTP 状态码 | 错误信息 | 触发场景 |
|-------------|---------|---------|
| 400 | `弹幕内容不能为空` | trim + 合并空白后内容为空 |
| 400 | `弹幕内容过长（最多80字）` | 内容超过 80 字符 |
| 400 | `弹幕内容包含敏感词，请修改后重试` | 命中敏感词库中的任一词 |
| 404 | `场次不存在` | sessionId 在数据库中无记录 |
| 404 | `用户不存在` | userId 在数据库中无记录 |
| 401 | - | JWT 缺失或无效（由 Guard 统一处理） |

## 10. 端到端流程

### 10.1 发送弹幕

```
用户输入弹幕内容
    ↓
    ├─ 内容为空 → 按钮禁用，无法发送
    ↓
    ├─ 点击"发送"或键盘确认
    │   ↓
    │   ├─ 前端：draft.trim() 为空 → 拦截
    │   ↓
    │   ├─ POST /api/danmaku { sessionId, content }
    │   │   ↓
    │   │   ├─ JWT 校验失败 → 401
    │   │   ├─ 内容为空 → 400
    │   │   ├─ 内容超长 → 400
    │   │   ├─ 命中敏感词 → 400
    │   │   ├─ 场次不存在 → 404
    │   │   └─ ✅ 入库成功 → 200 返回消息结构
    │   ↓
    │   ├─ 成功：清空输入框、更新 afterId、立即飘屏展示
    │   └─ 失败：全局 toast 提示错误信息
    ↓
    └─ 发送中（sending=true）：按钮禁用，防重复提交
```

### 10.2 增量拉取弹幕

```
页面加载 / sessionId 变化
    ↓
    ├─ 首次拉取（afterId=0）
    │   └─ GET /api/danmaku?sessionId=X&afterId=0&limit=50
    │      └─ 返回最近 50 条（时间正序）
    │          └─ 前端截取最后 8 条 → 飘屏展示
    ↓
    └─ 定时轮询（每 2.8s）
        └─ GET /api/danmaku?sessionId=X&afterId=maxId&limit=50
            └─ 返回 id > afterId 的新弹幕
                ├─ 有新消息 → 逐条飘屏展示
                └─ 无新消息 → 空列表，无操作
```

### 10.3 场次切换

```
sessionId 从 A 变为 B
    ↓
    ├─ 重置 afterId = 0
    ├─ 重置 initialPoll = true
    ├─ 清空飘屏（flying = []）
    ├─ 停止旧轮询
    └─ 立即发起首次拉取 + 开启新轮询
```

## 11. 非功能性约束

### 11.1 性能

| 场景 | 策略 |
|------|------|
| 高频轮询 | 2.8s 间隔 + 增量拉取（仅传新数据） |
| 大量弹幕 | `LIMIT 100` 上限 + 前端最多展示 16 条在屏 |
| 敏感词匹配 | 按词长降序、短词优先跳过，词库全内存加载 |

### 11.2 安全

| 场景 | 策略 |
|------|------|
| 未授权访问 | `JwtAuthGuard` 统一拦截 |
| 敏感内容 | 1000+ 条敏感词库 + 不区分 ASCII 大小写 |
| XSS 风险 | 小程序 `<text>` 组件天然防 XSS |
| 垃圾弹幕 | 80 字符上限 + 敏感词拦截 |

### 11.3 数据清理

- 场次删除时弹幕自动级联删除（`onDelete: 'CASCADE'`）
- 用户删除时弹幕自动级联删除

## 12. 当前设计的局限性

| 局限 | 说明 |
|------|------|
| 无频控 | 后端未实现单用户发送频率限制（如每分钟 N 条），依赖敏感词 + 80 字符上限做基础防护 |
| 无删除能力 | 管理员无法删除单条弹幕，无审核功能 |
| 无 WebSocket | 采用 HTTP 轮询（2.8s），非实时推送，可能有 2~3 秒延迟 |
| 无弹幕统计 | 无接口查询某场次弹幕总数、最活跃用户等 |
| 无楼层/回复 | 不支持弹幕回复、@用户、弹幕点赞等社交功能 |
| 无自定义样式 | 前端不支持弹幕颜色、字号、位置等自定义 |
| 无图片弹幕 | 仅支持纯文本 |
| 敏感词无热更新 | 词库仅在服务启动时加载，修改词库需重启服务（有 `reload()` 方法但未暴露接口） |
