# 竞猜系统 PRD（产品需求文档）

> **文档说明**：> **版本**：v1.0  
> **最后更新**：2026-05-11  
> **文档性质**：基于代码逆向梳理  

本文档基于 `Lumi_Sport_Space` 项目竞猜模块（`backend/src/modules/betting/`）的全部源码及前端页面逆向梳理，覆盖竞猜类型、赔率机制、虚拟币钱包、下注/结算/退款全链路。

---

## 一、模块概述

竞猜系统是 LumiSport 赛事互动平台中**业务最复杂**的子系统，面向观众用户，围绕线下赛事提供实时竞猜互动体验。观众使用「竞猜币」（虚拟币）对当前场次的多维度结果进行预测，猜中获得派奖，累积派奖总额决定「预言家段位」。

### 核心价值

| 维度 | 说明 |
|------|------|
| 用户价值 | 观众即使不上场也能深度参与赛事，增加等待/观赛期间的趣味性和粘性 |
| 运营价值 | 通过每日签到领币→下注→赢币→段位升级的正向循环驱动用户留存和日活 |
| 商业价值 | 为未来引入付费竞猜币、道具系统等商业化留出接口基础 |

## 二、竞猜类型体系

系统支持 **5 种竞猜类型**，每种对应不同的预测维度、适用玩法和赔率规则。

### 2.1 竞猜类型总览

| 竞猜类型 | 英文 Key | 适用玩法 | 预测内容 | 赔率类型 | 固定赔率 |
|----------|----------|----------|----------|----------|----------|
| **胜负竞猜** | `winBet` | 全部（冰球/拳王/击剑） | 猜红方或蓝方谁赢 | 动态赔率 | — |
| **元素之王** | `elementKing` | 冰球（大师模式） | 猜冰球/火球/风球哪种元素球数量最多 | 固定 | 3 倍 |
| **躺平之王** | `knockdownKing` | 拳王 | 猜本局双方总倒地次数 | 固定 | 3 倍 |
| **精准总分** | `preciseTotal` | 全部 | 猜双方得分之和（精确匹配） | 固定 | 20 倍 |
| **精准分差** | `preciseDiff` | 全部 | 猜双方得分之差的绝对值（精确匹配） | 固定 | 15 倍 |

### 2.2 胜负竞猜（winBet）

**核心玩法**：预测红方或蓝方哪一方获胜。

**适用场景**：所有玩法（冰球/拳王/击剑）均开放。

**参数**：
- `side`：必填，`red`（红方胜）或 `blue`（蓝方胜）
- `amount`：下注金额（≥ 10 竞猜币）

**赔率机制**（动态）：
```
赔率 = (1 / 预期胜率) × (1 - 手续费率)
其中：
  预期胜率 = 己方积分 / (己方积分 + 对方积分)
  手续费率 = 10%
  最终赔率钳位到 [1.1, 2.0]
```

**结算规则**：
| 比赛结果 | 用户猜中 | 用户未猜中 | 平局 |
|----------|----------|------------|------|
| 派奖 | 下注金额 × 对应赔率 | 0（损失本金） | 退还本金 |

### 2.3 元素之王（elementKing）

**核心玩法**：在冰球大师模式中，预测哪种元素球（冰球/火球/风球）数量最多。

**适用场景**：仅「疾速冰球 + 大师模式」开放。非大师模式或非冰球玩法时，下注会被拒绝。

**参数**：
- `choice`：必填，`ice`（冰球）/ `fire`（火球）/ `wind`（风球）
- `amount`：下注金额

**固定赔率**：3 倍

**结算规则**：
- 猜中（choice === elementWinner）：派奖 = 下注金额 × 3
- 未猜中：损失本金
- 未公布元素结果：记录「未公布元素结果」明细，不派奖不退还

### 2.4 躺平之王（knockdownKing）

**核心玩法**：预测拳王比赛中双方的总倒地次数。

**适用场景**：仅「烈焰拳王」玩法开放。

**参数**：
- `value`：必填，预测的倒地次数（整数 ≥ 0）
- `amount`：下注金额

**固定赔率**：3 倍

**结算规则**：
- 猜中（value === totalKnockdowns）：派奖 = 下注金额 × 3
- 未猜中：损失本金
- 未录入倒地次数：记录「未录入」明细

### 2.5 精准总分（preciseTotal）

**核心玩法**：精确预测双方得分的总和。

**适用场景**：所有玩法均开放。

**参数**：
- `value`：必填，预测的总分（整数 ≥ 0）
- `amount`：下注金额

**固定赔率**：20 倍

**结算规则**：
- 猜中（value === redScore + blueScore）：派奖 = 下注金额 × 20
- 未猜中：损失本金
- 比分未录入：记录「比分未录入」明细

### 2.6 精准分差（preciseDiff）

**核心玩法**：精确预测双方得分的绝对差值。

**适用场景**：所有玩法均开放。

**参数**：
- `value`：必填，预测的分差（整数 ≥ 0）
- `amount`：下注金额

**固定赔率**：15 倍

**结算规则**：
- 猜中（value === |redScore - blueScore|）：派奖 = 下注金额 × 15
- 未猜中：损失本金
- 比分未录入：记录「比分未录入」明细

## 三、竞猜币系统（虚拟币钱包）

### 3.1 概述

竞猜币是系统内的虚拟货币，用于竞猜下注。不与真实货币挂钩，仅用于平台内互动。

### 3.2 竞猜币获取方式

| 来源 | 数量 | 频次 | 说明 |
|------|------|------|------|
| **每日签到** | 100 币 | 每自然日一次（Asia/Shanghai 时区） | 调用 `POST /api/betting/daily-check-in` |
| **竞猜派奖** | 按赔率计算 | 每局结算时 | 猜中时获得：下注额 × 赔率 |
| **平局退还** | 等于下注额 | 每局结算时 | 胜负竞猜平局时退还本金 |
| **取消下注退还** | 等于下注额 | 竞猜阶段 | 取消已下注项，即时退还 |
| **流局/异常退还** | 等于下注额 | 管理员操作时 | 开赛流局/取消本局/踢人时全额退还 |

### 3.3 竞猜币消耗方式

| 去向 | 说明 |
|------|------|
| **竞猜下注** | 下注时即时扣除竞猜币 |
| **覆盖下注（增加金额）** | 修改已有下注为更高金额时，仅扣除差额 |

### 3.4 签到机制

```
签到判定：user.lastCheckInYmd !== today（Asia/Shanghai 自然日 YYYY-MM-DD）
奖励发放：user.coins += 100
记录更新：user.lastCheckInYmd = today
流水记录：coin_log 新增一条 kind='gain', reason='每日签到'
```

**幂等性**：同一天重复签到返回 `alreadyClaimedToday: true`，不重复发放。

### 3.5 竞猜币流水（coin_log）

每一笔竞猜币变动都会记录流水，包含：

| 字段 | 说明 |
|------|------|
| `kind` | `spend`（支出）或 `gain`（收入） |
| `amount` | 变动金额（整数） |
| `reason` | 原因描述（最长 500 字符），如「竞猜下注（赛事名 第1局）」 |
| `balanceAfter` | 变动后余额 |
| `sessionId` | 关联赛次（可为 null，如每日签到） |
| `createdAt` | 时间戳 |

## 四、下注流程

### 4.1 下注前置条件

观众用户下注需同时满足以下所有条件：

```
┌──────────────────────────────────────────────────────┐
│                   下注前置条件校验                      │
├──────────────────────────────────────────────────────┤
│ ① 场次存在（有未排除的最新场次）                        │
│ ② 场次状态为 BETTING（竞猜阶段）                       │
│ ③ 管理员已点击「开启竞猜」（guessingOpen = true）       │
│ ④ 红蓝双方选手均已就位（redUserId && blueUserId 非空）  │
│ ⑤ 用户未参赛（不是红方也不是蓝方选手）                  │
│ ⑥ 用户无待审核报名（无 pending 状态的 registration）   │
│ ⑦ 不存在实力悬殊（max(预期胜率) ≤ 80%）                │
│ ⑧ 竞猜币余额充足                                      │
│ ⑨ 下注金额 ≥ 10（最低下注额）                          │
│ ⑩ 竞猜类型与当前玩法匹配                               │
└──────────────────────────────────────────────────────┘
```

### 4.2 下注操作规则

| 规则 | 说明 |
|------|------|
| **每类型唯一** | 同一用户对同一场次的同一竞猜类型只能有一条下注记录（Unique 约束：sessionId + userId + betType） |
| **覆盖机制** | 对同一类型重复下注时，覆盖原有金额和选项。如新金额 > 旧金额，仅扣差额；如新金额 < 旧金额，退还多出部分 |
| **即时扣币** | 下注成功后即时扣除（或调整差额）竞猜币 |
| **事务保障** | 整个下注过程在数据库事务内完成，含悲观锁（`pessimistic_write`）防止并发问题 |

### 4.3 取消下注规则

| 规则 | 说明 |
|------|------|
| **时机** | 仅 `BETTING` 状态可取消 |
| **全额退还** | 取消时全额退还该条下注的竞猜币 |
| **删除记录** | 物理删除 session_bets 记录 |
| **流水记录** | 新增一条 `gain` 类型的流水，reason 为「取消下注退还」 |

### 4.4 各类型下注参数校验

| 竞猜类型 | 必填参数 | 玩法限制 | 校验规则 |
|----------|----------|----------|----------|
| winBet | side, amount | 无 | side ∈ {red, blue} |
| elementKing | choice, amount | 冰球 + 大师模式 | choice ∈ {ice, fire, wind} |
| knockdownKing | value, amount | 拳王 | value ≥ 0（整数） |
| preciseTotal | value, amount | 无 | value ≥ 0（整数） |
| preciseDiff | value, amount | 无 | value ≥ 0（整数） |

## 五、赔率机制

### 5.1 动态赔率（胜负竞猜）

胜负竞猜的赔率基于双方选手的积分实时计算：

```
步骤一：计算预期胜率
  红方胜率 = redRating / (redRating + blueRating)
  蓝方胜率 = blueRating / (redRating + blueRating)

步骤二：计算原始赔率
  红方赔率_原始 = (1 / 红方胜率) × (1 - 0.1)    // 10% 手续费
  蓝方赔率_原始 = (1 / 蓝方胜率) × (1 - 0.1)

步骤三：钳位到合法区间
  最终赔率 = clamp(原始赔率, 1.1, 2.0)    // 最低 1.1 倍，最高 2.0 倍

步骤四：四舍五入到两位小数
```

**赔率展示规则**：
- **报名/竞猜阶段**：实时显示（随选手积分变化而变化）
- **开赛（started）及之后**：展示锁定值（`lockedRedOdds` / `lockedBlueOdds`，在管理员点击「开局」时写入）
- **无选手时**：使用默认积分 100 计算

### 5.2 固定赔率（特殊竞猜）

| 竞猜类型 | 赔率 | 说明 |
|----------|------|------|
| 元素之王 | 3× | 猜中哪方元素球最多 |
| 躺平之王 | 3× | 精确猜中总倒地次数 |
| 精准总分 | 20× | 精确猜中双方得分总和 |
| 精准分差 | 15× | 精确猜中双方得分之差 |

### 5.3 赔率锁场

管理员点击「开局」时：
1. 读取当前红蓝双方在该玩法的积分
2. 按上述公式计算赔率
3. 写入 `game_sessions.locked_red_odds` 和 `locked_blue_odds`
4. 此后竞猜状态接口返回的 `odds` 固定为锁场值

## 六、竞猜开放控制

### 6.1 开放/关闭条件矩阵

竞猜是否可下注取决于多重条件的组合：

| 条件 | winBetEnabled | 说明 |
|------|---------------|------|
| 场次状态 ≠ BETTING | ❌ | 不在竞猜阶段 |
| guessingOpen = false | ❌ | 管理员未点「开启竞猜」 |
| 双方未齐人 | ❌ | 红方或蓝方无选手 |
| 实力悬殊（max 胜率 > 80%） | ❌ | 防止不公平竞猜 |
| 以上条件全满足 | ✅ | 可正常下注 |

### 6.2 竞猜提示文案（preStartBettingHint）

根据当前状态，前端展示对应的提示信息：

| 状态 | 提示文案 |
|------|----------|
| 双方未齐人 | 「当前为报名阶段；待红蓝双方选手均通过审核就位后，将开放观众竞猜。」 |
| 管理员未开竞猜 | 「双方选手已定；请管理员在「场次管理」流程操作中点击「开启竞猜」后，观众方可下注。」 |
| 实力悬殊 | 「红蓝均已就位，但实力悬殊过大（参考：任一侧预期胜率高于 80%），本场暂不开放竞猜下注」 |
| 正常开放 | 「竞猜已开放：下注后请以开赛结果为准（若临场不齐人等，开赛时仍可流局作废并退还）。」 |

### 6.3 实力悬殊判定

```
触发条件：max(红方预期胜率, 蓝方预期胜率) > 80%
效果：
  - 竞猜阶段：禁止下注（前端不可点击，后端拒绝请求）
  - 开赛瞬间：自动流局，所有已下注全额退还
```

## 七、结算系统

### 7.1 结算时机

场次状态从 `started` 变为 `settled` 时（管理员录入最终比分后），在同一数据库事务内完成竞猜结算。

### 7.2 结算流程

```
┌─────────────────────────────────────────────────────┐
│                   竞猜结算流程                        │
├─────────────────────────────────────────────────────┤
│                                                     │
│  1. 读取本局所有 session_bets                         │
│     ↓                                               │
│  2. 如无下注记录，直接返回                             │
│     ↓                                               │
│  3. 解析锁定赔率（优先 locked_odds，否则实时计算）      │
│     ↓                                               │
│  4. 按用户分组                                       │
│     ↓                                               │
│  5. 逐条计算每条 bet 的结算结果：                      │
│     ├─ 胜负竞猜 → 按锁定赔率结算                      │
│     ├─ 元素之王 → 固定 3x                            │
│     ├─ 躺平之王 → 固定 3x                            │
│     ├─ 精准总分 → 固定 20x                           │
│     └─ 精准分差 → 固定 15x                           │
│     ↓                                               │
│  6. 按用户汇总：                                     │
│     ├─ totalBetted（总下注本金）                      │
│     ├─ totalWonStat（猜中项派奖合计，用于段位统计）     │
│     ├─ totalPayout（实际发放币 = 猜中派奖 + 平局退还）  │
│     └─ netResult（净盈亏 = totalWonStat + 退还 - 本金）│
│     ↓                                               │
│  7. 派奖：user.coins += totalPayout                   │
│     ↓                                               │
│  8. 累计统计：user.totalBettingWinnings += totalWonStat│
│     ↓                                               │
│  9. 写入结算记录：bet_settlement_records              │
│     ↓                                               │
│  10. 写入竞猜币流水                                  │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 7.3 结算明细结构

每条结算记录包含 `detailsJson`（JSON 字符串），记录每个竞猜类型的明细：

```json
[
  {
    "type": "胜负竞猜",
    "myBet": "红方胜",
    "amount": 100,
    "actual": "红方胜",
    "result": "猜中",
    "won": 150
  },
  {
    "type": "精准总分",
    "myBet": "5 分",
    "amount": 50,
    "actual": "5 分",
    "result": "猜中",
    "won": 1000
  }
]
```

**result 取值**：
| result | 含义 | 金额处理 |
|--------|------|----------|
| `猜中` | 预测正确 | `won` = 下注额 × 赔率 |
| `未猜中` | 预测错误 | `lost` = 下注额（本金损失） |
| `平局` | 胜负竞猜平局 | `refund` = 下注额（退还本金） |
| `比分未录入` / `未公布元素结果` / `未录入倒地次数` | 管理员未填写完整数据 | 不派奖不退还 |

### 7.4 关键统计指标区分

| 指标 | 用途 | 计算方式 |
|------|------|----------|
| `totalPayout` | 实际发到用户账户的币 | 猜中派奖 + 平局退还，四舍五入 |
| `totalWonStat`（累计竞猜派奖） | 预言家段位升级依据 | **仅**猜中项的派奖（不含平局退还），更新 `user.totalBettingWinnings` |
| 
etResult` | 本局净盈亏 | `totalWonStat + refund - totalBetted` |

## 八、退款场景

系统支持 5 种退款场景，各有不同触发时机和副作用：

| 场景 | 触发方 | 退款后副作用 | 说明 |
|------|--------|-------------|------|
| **开赛流局** | 系统自动（开局校验不通过） | 标记 `betsVoidedAtMatchStart=true`，比赛继续进行 | 双方不齐人或实力悬殊 |
| **取消本局** | 管理员 | 局数+1、状态→WAITING、清空比分/选手/赔率、删除报名记录 | 完全重置本局 |
| **踢出选手** | 管理员 | 关闭竞猜、清空锁定赔率、状态保持 BETTING | 需重新报名 |
| **强制废弃** | 管理员（调试） | 标记 `excluded_from_latest` | 不改局数和状态 |
| **用户取消下注** | 观众 | 物理删除该条 bet 记录 | 仅限 BETTING 阶段 |

**退款统一流程**：
1. 查询本场所有 `session_bets`
2. 按用户汇总应退金额
3. `user.coins += 退款金额`
4. 写入 `coin_log`（kind=gain）
5. 删除所有 `session_bets`（物理删除）

## 九、预言家段位系统

### 9.1 段位等级

段位根据用户的 `totalBettingWinnings`（累计竞猜派奖）自动判定：

| 段位 | 图标 | 累计派奖阈值 |
|------|------|-------------|
| 新手预言家 | 🔮 | ≥ 0 |
| 青铜预言家 | 🥉 | ≥ 2,500 |
| 白银预言家 | 🥈 | ≥ 5,000 |
| 黄金预言家 | 🥇 | ≥ 10,000 |
| 钻石预言家 | 💎 | ≥ 20,000 |
| 铂金预言家 | 👑 | ≥ 50,000 |
| 王者预言家 | 🏆 | ≥ 100,000 |

### 9.2 段位计算规则

```
输入：totalBettingWinnings（累计竞猜派奖，仅猜中项）
算法：从低到高遍历阈值，取满足条件的最高段位
更新时机：每局结算时实时更新
```

> **注意**：段位**只升不降**——平局退还不计入 `totalBettingWinnings`，但猜中派奖一旦计入永不减少。

## 十五、端到端业务流程

### 15.1 标准竞猜流程（正常流程）

```
管理员创建场次 → 管理员开启竞猜 → 双方选手报名就位
                    ↓
        观众查看赔率，选择竞猜类型下注
                    ↓
        管理员点击「开局」→ 锁定赔率
                    ↓
            比赛进行中（不可下注）
                    ↓
        管理员录入比分 → 自动结算
                    ↓
    猜中者获派奖 → 段位可能升级 → 流水记录写入
```

### 15.2 实力悬殊流程

```
观众已下注 → 开赛时校验 max(胜率) > 80%
                    ↓
        全部押注作废 → 全额退还竞猜币
                    ↓
        标记 betsVoidedAtMatchStart = true
                    ↓
        前端展示流局横幅提示
```

### 15.3 取消本局流程

```
管理员在竞猜阶段点击「取消本局」
                    ↓
    全额退还所有下注 → 删除 session_bets
                    ↓
    局数+1 → 状态→WAITING → 清空选手/比分/赔率
                    ↓
        删除所有报名记录
```

## 十六、并发安全设计

| 场景 | 策略 |
|------|------|
| 并发下注 | 场次行级悲观锁（`pessimistic_write`）+ 事务 |
| 并发签到 | 用户行级悲观锁 + `lastCheckInYmd` 幂等判定 |
| 结算竞态 | 在赛事结算同一事务内完成，天然串行 |
| 覆盖下注 | 唯一约束 `UQ_session_user_bet_type` + 事务内 upsert |

## 十七、异常处理与边界情况

| 异常情况 | 处理方式 |
|----------|----------|
| 比分未录入即结算 | 记录「比分未录入」明细，不派奖不退还 |
| 元素结果未公布 | 记录「未公布元素结果」明细 |
| 倒地次数未录入 | 记录「未录入倒地次数」明细 |
| 平局（胜负竞猜） | 退还本金，不计入 `totalWonStat` |
| 用户参赛后不可竞猜 | 前端显示警告，后端拒绝下注请求 |
| 待审核报名期间 | 不可竞猜（防止报名被拒后竞猜数据残留） |
| 管理端大盘池无场次 | 返回空结构占位（sessionId=0），不报错 |
