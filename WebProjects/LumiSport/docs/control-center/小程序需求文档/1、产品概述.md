## 2. 产品定位

**LumiSport Space** 是一款面向 **线下运动竞技门店** 的 **赛事互动平台**，以微信小程序为核心载体，为门店中的 **选手（玩家）** 和 **观众** 提供沉浸式的竞技参与和观赛竞猜体验，同时为 **管理员（店员）** 提供轻量化的赛事管控工具。

### 2.1 一句话定位

> 线下运动竞技场馆的赛事管理 + 观众互动竞猜平台——让每场比赛都有悬念。

### 2.2 核心价值主张

| 角色 | 价值 |
|------|------|
| **选手** | 报名参赛、实时对战、积分排名，感受竞技成长的成就感 |
| **观众** | 竞猜下注、弹幕互动、段位成长，即使在场下也能深度参与 |
| **管理员** | 一站式场次管理（创建→报名→开赛→结算→下一轮），掌控全局节奏 |

## 3. 目标用户与角色体系

### 3.1 用户角色

系统定义两种角色（对应后端 `UserRole` 枚举）：

| 角色 | 标识 | 说明 | 入口 |
|------|------|------|------|
| **普通用户（选手/观众）** | `user` | 通过微信登录，可报名参赛、观看赛事、参与竞猜、发送弹幕 | 微信小程序 |
| **管理员** | `admin` | 通过管理员手机号免密登录，拥有场次管理的全部权限 | 微信小程序管理端页面 |

### 3.2 同一用户的多重身份

一个用户在不同场次中可以扮演不同身份：

- **参赛选手**：报名并进入红方/蓝方后，本场不可竞猜（避免利益冲突）
- **观众（竞猜者）**：未参赛时可作为观众对比赛结果进行竞猜
- **管理员**：拥有赛事管控权限，不参与竞猜

### 3.3 登录方式

| 方式 | 对象 | 实现细节 |
|------|------|----------|
| **微信一键登录** | 普通用户 | 调用 `wx.login` 获取 `code` → 后端 `jscode2session` 获取 `openid` → 自动注册或匹配已有账户 → 签发 JWT |
| **管理员免密登录** | 管理员 | 输入预配置的管理员手机号 → 后端校验手机号匹配 → 签发 JWT（含 `role: admin`） |

## 5. 场次生命周期（核心状态机）

每个场次（`GameSession`）遵循严格的状态机流转，管理员按步骤推进：

```
┌─────────┐     管理员配置      ┌─────────┐    管理员开始报名    ┌──────────┐
│  (无场次) │ ──────────────→ │ waiting │ ──────────────→ │ betting  │
└─────────┘                  └─────────┘                  └──────────┘
                                  ↑                          │
                                  │   管理员取消本局           │ 管理员开赛
                                  │   （全额退款，局数+1）      │
                                  └────────────────────── ┌──────────┐
                                                          │ started  │
                                                          └──────────┘
                                                               │
                                                        管理员录入比分
                                                               │
                                                          ┌──────────┐
                                                          │ settled  │
                                                          └──────────┘
                                                               │
                                                        管理员开启下一轮
                                                               │
                                                          ┌──────────┐
                                                          │ waiting  │ (新局)
                                                          └──────────┘
```

### 5.1 各状态说明

| 状态 | 枚举值 | 可执行操作 | 说明 |
|------|--------|------------|------|
| **等待配置** | `waiting` | 管理员：配置玩法/名称、开始报名 | 初始状态或上一局结算后新建的空白局 |
| **报名中** | `betting` | 选手：报名/撤销参赛；管理员：审核报名、开启竞猜、开赛、取消本局 | 双重含义阶段——既允许选手报名，也允许观众竞猜下注 |
| **比赛中** | `started` | 管理员：录入比分、结束比赛 | 比赛进行中，竞猜已锁定赔率，不可再下注或取消 |
| **已结算** | `settled` | 管理员：开启下一轮、返回首页 | 比分已录入，竞猜已结算派奖，积分已更新 |

### 5.2 局数（Round）机制

- 每个场次包含多局，`round` 字段从 1 开始递增
- 管理员点击"开启下一轮"时：保留当前 `game_sessions` 记录，`round + 1`，状态回到 `waiting`
- 管理员点击"取消本局"时：全额退还竞猜押注，`round + 1`，状态回到 `waiting`

## 7. 竞猜系统详解

竞猜系统是本产品 **业务最复杂、逻辑最密集** 的核心模块，支撑观众在观赛时的深度参与感。

### 7.1 五种竞猜类型

| 类型 | 标识 | 适用玩法 | 赔率 | 说明 |
|------|------|----------|------|------|
| **胜负竞猜** | `winBet` | 全部玩法 | 动态赔率（1.1~2.0） | 预测红方或蓝方获胜；赔率由双方积分实时计算 |
| **元素之王** | `elementKing` | 仅冰球大师模式 | 固定 3x | 预测冰/火/风哪种元素球出现最多 |
| **躺平之王** | `knockdownKing` | 仅拳王 | 固定 3x | 精准预测双方倒地总次数 |
| **精准总分** | `preciseTotal` | 全部玩法 | 固定 20x | 精准预测双方得分之和 |
| **精准分差** | `preciseDiff` | 全部玩法 | 固定 15x | 精准预测双方得分之差的绝对值 |

### 7.2 赔率机制

- **胜负赔率**：基于双方选手的当前玩法积分（`rating`）动态计算
  - 计算公式：`赔率 = (1 / 预期胜率) × (1 - 10% 手续费)`，范围 `[1.1, 2.0]`
  - 预期胜率：`myRating / (myRating + enemyRating)`
  - **锁定时机**：开赛瞬间（`betting → started`）锁定赔率快照，赛后结算以锁定值为准
- **固定赔率**：元素之王 3x、躺平之王 3x、精准总分 20x、精准分差 15x

### 7.3 虚拟货币体系

| 项目 | 规则 |
|------|------|
| **货币名称** | 竞猜币（coins） |
| **初始额度** | 新用户注册时为 0 |
| **每日签到** | 每日可签到一次，领取 **100** 竞猜币（按 Asia/Shanghai 自然日） |
| **下注最低额** | 单次下注最低 **10** 竞猜币 |
| **消费场景** | 对各竞猜类型下注 |
| **获取场景** | 签到、竞猜猜中派奖、流局/平局退还 |

### 7.4 竞猜开放规则

竞猜并非始终开放，需满足一系列前置条件：

| 条件 | 说明 |
|------|------|
| 场次处于 `betting` 状态 | 已过 waiting，进入报名阶段 |
| 管理员已手动"开启竞猜" | `guessingOpen = true` |
| 双方选手均已就位 | 红蓝方均有正式报名选手 |
| 实力未悬殊 | 双方预期胜率中较高者 **不超过 80%** |
| 用户未参赛 | 当前用户不是本场红方或蓝方选手 |
| 用户未在待审列表 | 当前用户不在任一方待审报名中 |
| 对手非机器人 | 非机器人对手场景 |

**违规场景自动处理**：
- **开赛流局**：开赛瞬间若未齐人或实力悬殊（>80%），本场所有押注作废并全额退还
- **实力悬殊封盘**：报名阶段若检测到实力差距过大，自动暂停竞猜下注

### 7.5 结算规则

| 竞猜类型 | 猜中 | 未猜中 | 平局 |
|----------|------|--------|------|
| 胜负竞猜 | 返还 `下注额 × 锁定赔率` | 不返还 | 返还下注额 |
| 元素之王 | 返还 `下注额 × 3` | 不返还 | — |
| 躺平之王 | 返还 `下注额 × 3` | 不返还 | — |
| 精准总分 | 返还 `下注额 × 20` | 不返还 | — |
| 精准分差 | 返还 `下注额 × 15` | 不返还 | — |

> **注意**：平局退还仅适用于胜负竞猜；其他类型不存在"平局"概念。

## 8. 积分与排名系统

### 8.1 选手积分

- 每个用户在每种玩法下拥有 **独立积分**（`user_game_ratings` 表）
- 初始积分：**100 分**，最低不低于 **1 分**
- **仅双方均存在且非平局时**更新积分
- 积分变化量：`Δ = 20 × (对手积分 / 双方积分之和)`
  - 胜者加分，败者扣分
  - 双方实力越接近，变化量越接近（10, 10）；差距越大，变化量越悬殊
- 平局：双方积分不变

### 8.2 排行榜

- 按玩法维度独立排名
- 展示用户昵称和当前积分

## 11. 系统架构概览

### 11.1 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| **客户端** | uni-app + Vue 3 + Pinia + TypeScript | 编译为微信小程序（主要）和 H5（兼容） |

### 11.2 子仓库关系

```
LumiSport/
├── Lumi_Sport_Space/      # 小程序 + 后端 API（核心仓库）
│   ├── backend/           # NestJS 后端
│   └── uniapp/            # uni-app 前端
├── Server/                # Unity 大屏服务端
├── SportClient/           # Unity 大屏客户端（YVR）
└── docs/                  # 跨仓库共享文档
```

### 11.3 大屏与小程序的协作

- **Server（大屏后端）** 与 **NestJS（小程序后端）** 共享同一份场次数据
- 大屏端负责玩法选择与实时赛况展示
- 小程序管理端负责场次配置、报名审核、比分录入等管控操作
- 双端操作需通过版本号（`configEpoch`）机制解决并发冲突
- 详见《大屏与小程序管理端-场次同步》设计文档

## 13. 端到端业务流程

### 13.1 完整赛事流程（管理员视角）

```
1. 管理员登录小程序
2. 进入「场次管理」→ 配置玩法（冰球/拳王/击剑）、名称、是否大师模式
3. 点击「开始报名」→ 场次进入 betting 状态
4. 选手在「我要参赛」页选择红方或蓝方报名
5. 管理员审核报名 → 通过后该侧锁定
6. 红蓝双方就位后 → 管理员点击「开启竞猜」→ 观众可下注
7. 观众在「我要竞猜」页选择竞猜类型并下注
8. 管理员点击「正式开始比赛」→ 场次进入 started 状态
   └─ 系统自动：锁定赔率、校验是否流局（未齐人/悬殊则退款）
9. 比赛结束 → 管理员录入比分（含扩展字段：冰火风球/倒地次数）
10. 系统自动：结算竞猜派奖、更新选手积分
11. 管理员点击「开启下一轮」→ 新局开始
```

### 13.2 观众竞猜流程

```
1. 微信登录小程序
2. 进入「赛事」首页 → 查看当前场次信息
3. 每日签到领取 100 竞猜币（如当日未签到）
4. 进入「我要竞猜」→ 查看赔率与各竞猜类型
5. 选择竞猜类型 → 输入下注金额 → 确认下注
6. 可在开赛前取消下注（退还竞猜币）
7. 比赛结束后 → 查看结算结果
8. 在「我的竞猜」查看汇总、「竞猜排行」查看排名
```

### 13.3 选手参赛流程

```
1. 微信登录小程序
2. 进入「我要参赛」→ 选择红方或蓝方 → 提交报名
3. 等待管理员审核通过
4. 通过后可在「赛事」页查看自己的阵营
5. 比赛结束后查看结果与积分变化
6. 在「我的」页查看各玩法积分、历史对局
```

## 14. 关键业务规则汇总

| 编号 | 规则 | 说明 |
|------|------|------|
| BR-01 | **参赛者不可竞猜** | 已报名红方/蓝方的用户，本场所有竞猜类型均不可下注 |
| BR-02 | **待审报名者不可竞猜** | 已提交报名但尚未通过审核的用户同样不可竞猜 |
| BR-03 | **开赛流局** | 开赛瞬间若未齐人或任一方预期胜率 >80%，本场所有押注作废并全额退还 |
| BR-04 | **实力悬殊封盘** | 报名阶段若检测到实力差距过大（>80%），自动暂停竞猜下注 |
| BR-05 | **赔率锁定** | 开赛瞬间锁定当前赔率快照，结算以锁定值为准 |
| BR-06 | **平局退还** | 胜负竞猜出现平局时，退还下注额（其他类型无平局概念） |
| BR-07 | **单类型唯一** | 同一用户在同一场次的同一竞猜类型只能有一条下注记录（可覆盖或取消） |
| BR-08 | **局数递增** | 取消本局或开启下一轮时，`round` 字段 +1 |
| BR-09 | **积分最小值** | 用户积分最低为 1 分，不会降到 0 或负数 |
| BR-10 | **签到日切** | 签到按 Asia/Shanghai 自然日计算，每日一次 |
| BR-11 | **竞猜开放需管理员手动开启** | 即使双方已就位且实力均衡，也需管理员点击"开启竞猜"才开放 |
| BR-12 | **退款安全** | 流局退款、取消本局退款、管理员踢出选手退款均使用悲观锁保证余额一致性 |

## 15. 风险与合规

| 领域 | 说明 |
|------|------|
| **虚拟货币** | 竞猜币为纯虚拟货币，不涉及真实货币兑换，需确保小程序类目与内容审核合规 |
| **UGC 内容** | 弹幕为用户生成内容，需敏感词过滤，满足平台内容规范 |
| **防刷与风控** | 下注频控、弹幕频控、账户锁定机制，防止恶意刷单 |
| **数据安全** | 所有结算与积分更新均在服务端完成，客户端仅展示，不信任前端计算 |
| **并发安全** | 竞猜币余额更新使用数据库悲观锁，防止超扣 |

## 16. 术语表

| 术语 | 含义 |
|------|------|
| **场次（Session）** | 一场完整的比赛活动，包含多局（Round） |
| **局（Round）** | 场次内的一轮比赛，从报名到结算为完整一局 |
| **大师模式（Master Mode）** | 冰球的扩展玩法，需录入冰/火/风球出现次数 |
| **竞猜币（Coins）** | 平台虚拟货币，用于竞猜下注 |
| **预期胜率** | 基于双方积分计算的胜率预估值 `myRating / (myRating + enemyRating)` |
| **锁定赔率** | 开赛瞬间快照的赔率，结算以此为准 |
| **流局（Void）** | 因条件不满足（未齐人/实力悬殊）导致本场押注作废并退款 |
| **预言家段位** | 根据累计竞猜派奖划分的等级，从新手到王者共 7 级 |
| **竞猜大盘** | 管理员可查看的当前场次所有人下注汇总 |
| **红方/蓝方** | 比赛的双方阵营，选手报名时选择 |
